**Purpose**: Thorough requirements quality validation across CLI contract, security, API coverage, and agent integration — unit tests for the English specification
**Created**: 2026-03-14
**Feature**: [spec.md](../spec.md)
**Depth**: Thorough
**Audience**: Reviewer (pre-implementation gate)
**Focus**: CLI Contract & Agent Integration, Security & Credential Management, API Coverage & Error Handling

## Requirement Completeness

- [x] CHK001 Are the specific fields displayed in human-readable output defined for each command, or only implied? (e.g., `bb pr list` says "titles, authors, status, and IDs" but `bb pipeline list` says "run IDs, trigger info, status, and timestamps" — field lists vary in specificity) [Completeness, Spec §US2 vs §US7] — **Resolved**: FR-009, FR-010, FR-011, FR-036, FR-041 now explicitly enumerate displayed fields for human-readable output.
- [x] CHK002 Are requirements defined for what happens when `--workspace` is not set via flag, config, or environment variable and a command that requires it is run? [Gap, Spec §FR-004a] — **Resolved**: FR-004a now specifies the exact error message including guidance to use --workspace flag, bb auth setup, or BITBUCKET_WORKSPACE env var.
- [x] CHK003 Are the exact environment variable names for OAuth2 token override documented in the spec? (BITBUCKET_ACCESS_TOKEN is only mentioned in data-model.md, not in spec.md FR-003a) [Completeness, Spec §FR-003a] — **Resolved**: FR-003a now explicitly lists all four env var names: BITBUCKET_USERNAME, BITBUCKET_APP_PASSWORD, BITBUCKET_ACCESS_TOKEN, BITBUCKET_WORKSPACE.
- [x] CHK004 Is a requirement defined for what `bb auth status` displays when credentials come from environment variables vs config file? [Gap, Spec §US1] — **Resolved**: New FR-005a defines bb auth status output including credential source indication ("Source: environment variables" or "Source: config file") and masked credential values.
- [x] CHK005 Are requirements defined for the `--verbose` flag mentioned in the CLI contract but absent from the functional requirements in spec.md? [Gap, Contract vs Spec §FR] — **Resolved**: New FR-008a defines --verbose behavior: HTTP request/response details to stderr with masked auth headers.
- [x] CHK006 Are requirements specified for `bb pr update` behavior when no update flags are provided (no --title, --description, --reviewer, --destination)? [Gap, Spec §FR-014] — **Resolved**: FR-014 now specifies error message when no update flags provided.
- [x] CHK007 Is the merge commit message behavior specified when `--message` is not provided to `bb pr merge`? (Default from Bitbucket vs empty?) [Gap, Spec §FR-018] — **Resolved**: FR-018 now states "When --message is not provided, the Bitbucket API uses its default merge commit message."
- [x] CHK008 Are requirements defined for what happens when a user tries to decline an already-merged PR or approve a declined PR? [Gap, Spec §Edge Cases] — **Resolved**: Edge cases section now includes both scenarios: CLI relays API errors for declining merged PRs and approving non-open PRs.
- [x] CHK009 Is a requirement specified for config file creation behavior on first `bb auth setup` — does it create the directory and file if they don't exist? [Gap, Spec §FR-003] — **Resolved**: FR-003 now states "If bb auth setup is run and the directory or file does not exist, they MUST be created automatically."
- [x] CHK010 Are requirements defined for `bb pipeline run` when the repository has no `bitbucket-pipelines.yml`? [Gap, Spec §US7] — **Resolved**: FR-037 now states the CLI relays the error from the Bitbucket API when no pipeline config exists.
- [x] CHK011 Is the `--page` flag mentioned in FR-044 but absent from all command contracts — is page-based navigation actually in scope? [Conflict, Spec §FR-044 vs Contract] — **Resolved**: FR-044 updated to specify --limit and --page-size only. No --page flag. The CLI auto-paginates via next links.

## Requirement Clarity

- [x] CHK012 Is "clear error message" quantified with specific content requirements? FR-005 says "including guidance on how to fix the issue" — are the exact guidance messages or their structure defined? [Clarity, Spec §FR-005] — **Resolved**: FR-005 now specifies error message structure: (a) what went wrong, (b) how to fix it, (c) scope guidance for scope-related errors.
- [x] CHK013 Is "human-readable text output" (FR-006) specified with a concrete format? (Tabular? Key-value pairs? Indented? Column widths?) The contract defines "tabular for lists, key-value for details" but the spec itself doesn't. [Clarity, Spec §FR-006 vs Contract] — **Resolved**: FR-006 now specifies: tabular for lists, key-value for details, plain text unified format for diffs.
- [x] CHK014 Is "filtering and sorting" for `bb repo list` (FR-009) defined with specific supported filter fields, or is it left to the API's capability? [Clarity, Spec §FR-009] — **Resolved**: FR-009 now specifies --role filter (admin, contributor, member, owner) and --sort flag, with values passed directly to the API.
- [x] CHK015 Are the "detailed metadata" fields for `bb repo view` (FR-010) explicitly enumerated in the spec, or only in the acceptance scenario of US8? [Clarity, Spec §FR-010] — **Resolved**: FR-010 now explicitly lists all displayed fields: full name, slug, description, language, default branch, dates, size, privacy, URL.
- [x] CHK016 Is "handle long-running merge responses (202 with task status)" (FR-018) specified with what exactly the CLI outputs? (Task ID? Polling URL? Instruction text?) [Clarity, Spec §FR-018] — **Resolved**: FR-018 now specifies exact output: "Merge in progress. Task ID: {id}. Check PR status to confirm completion."
- [x] CHK017 Is "standard config directory" in FR-003 defined with per-platform paths? (`~/.config/` on Linux, `%APPDATA%` on Windows, `~/Library/` on macOS?) [Clarity, Spec §FR-003] — **Resolved**: FR-003 now specifies platform-specific paths for Linux, macOS, and Windows.
- [x] CHK018 Are the reviewer identifiers for `--reviewer` flag specified — username, UUID, display name, or account ID? [Clarity, Spec §FR-012] — **Resolved**: FR-012 now specifies reviewers are identified by Bitbucket username (nickname).
- [x] CHK019 Is "pipeline step details" (FR-041) defined with what specific information beyond name and status is shown? [Clarity, Spec §FR-041] — **Resolved**: FR-041 now lists: step name, UUID, status, start/completion time, duration, log URL.

## Requirement Consistency

- [x] CHK020 Do the user story acceptance scenarios and functional requirements align on PR update behavior? US3 says "updates its title or description" but FR-014 adds "reviewers, destination branch" — is US3 acceptance scenario incomplete? [Consistency, Spec §US3 vs §FR-014] — **Resolved**: US3 acceptance scenario 3 updated to include "title, description, reviewers, or destination branch."
- [x] CHK021 Is the credential resolution order consistent between spec.md (FR-003a: "env vars take precedence over config file") and the plan.md (env → config file)? Both agree, but is flag-level override (--username on command) also in scope or explicitly excluded? [Consistency, Spec §FR-003a vs Contract] — **Resolved**: FR-003a now explicitly states "No per-command flag-level credential override is supported."
- [x] CHK022 Is the pagination flag naming consistent? FR-044 says `--limit, --page` but the CLI contract defines `--limit, --page-size`. Is `--page` for page number navigation or `--page-size` for controlling page size — or both? [Consistency, Spec §FR-044 vs Contract] — **Resolved**: FR-044 updated to --limit and --page-size only. --page removed. Consistent with contract.
- [x] CHK023 Are exit codes consistent across all commands? The contract defines only 0 and 1, but different error types (auth, not-found, rate-limit, input validation) all map to exit code 1 — is differentiation intentionally excluded? [Consistency, Contract §Exit Codes] — **Resolved**: FR-046 and contract now explicitly state single error exit code 1 is intentional. Error differentiation is via messages/JSON, not exit codes.
- [x] CHK024 Is `bb pr activity` (FR-021) consistently placed? It's in US2 (Browse and Manage PRs) as an implementation task, but conceptually belongs with US5 (View PR Diffs, Commits, and Activity). Are the user story boundaries clearly delineated? [Consistency, Spec §US2 vs §US5] — **Resolved**: Accepted as-is. FR-021 is in the functional requirements independently of user story placement. US5 covers "View PR Diffs, Commits, and Activity" and references the activity log. The task ordering in tasks.md places activity implementation with US5.
- [x] CHK025 Does the Task entity description in Key Entities ("Has completion status") align with FR-035 which sources tasks from the activity log? Are task fields (description, assignee, status) actually available from the activity log endpoint? [Consistency, Spec §Key Entities vs §FR-035] — **Resolved**: Task entity and FR-035 updated to note that assignee may not be available from the activity log. Available fields limited to description and completion status.

## Acceptance Criteria Quality

- [x] CHK026 Is SC-001 ("within 2 minutes of installation") measurable given that installation steps aren't defined in the spec? Does the 2-minute window include pnpm install time? [Measurability, Spec §SC-001] — **Resolved**: SC-001 clarified to start after pnpm install and build. 2-minute window covers auth setup + first command only.
- [x] CHK027 Is SC-002 ("identical outcomes") objectively verifiable? How is "identical" defined when the CLI shows a subset of fields vs the full Bitbucket web UI? [Measurability, Spec §SC-002] — **Resolved**: SC-002 changed from "identical" to "equivalent" with clarification that API state changes are the same; CLI may display a subset of fields.
- [x] CHK028 Is SC-003 ("within 10 seconds") measurable given the caveat "excluding network latency beyond the tool's control"? What constitutes "the tool's control" vs external latency? [Measurability, Spec §SC-003] — **Resolved**: SC-003 clarified: "tool's control" = CLI processing time (parsing, formatting), not network round-trip or API response time.
- [x] CHK029 Is SC-004 (agent workflow: create PR, add comment, approve, merge) complete enough to validate end-to-end? Does it need to include error recovery scenarios? [Measurability, Spec §SC-004] — **Resolved**: SC-004 updated to include error recovery: JSON error output must provide sufficient information for agent diagnosis.
- [x] CHK030 Can SC-006 ("consistent in structure") be objectively tested without a JSON schema definition? Is a formal JSON schema for each command's output in scope? [Measurability, Spec §SC-006] — **Resolved**: SC-006 clarified with specific structure patterns. Formal JSON schema is out of scope for v1; patterns documented in contract.

## Scenario Coverage

- [x] CHK031 Are requirements defined for the scenario where both `--username`/`--app-password` AND `--access-token` are provided to `bb auth setup`? Which takes precedence, or is it an error? [Coverage, Spec §FR-001/FR-002] — **Resolved**: FR-004 now specifies this is an error: "the CLI MUST reject the input with an error explaining that only one authentication method should be specified."
- [x] CHK032 Are requirements defined for the scenario where environment variables provide partial credentials (e.g., BITBUCKET_USERNAME set but BITBUCKET_APP_PASSWORD not set)? [Coverage, Spec §FR-003a] — **Resolved**: FR-003a now specifies: "CLI MUST display an error listing which variables are set and which are missing."
- [x] CHK033 Are requirements defined for comment reply threading? The contract includes `--parent` flag, but no acceptance scenario covers creating reply comments. [Coverage, Spec §US4 vs Contract] — **Resolved**: New FR-027a added defining --parent flag for threaded replies.
- [x] CHK034 Are requirements defined for `bb pr list` filtering by multiple states simultaneously? (e.g., OPEN and MERGED together?) [Coverage, Spec §FR-011] — **Resolved**: FR-011 now specifies "Only a single state filter is supported per invocation."
- [x] CHK035 Are requirements defined for the scenario where `bb pr publish` is called on a non-draft PR? [Coverage, Spec §FR-023] — **Resolved**: FR-023 now specifies no-op behavior: display "PR #{id} is already published" and exit with code 0.
- [x] CHK036 Are requirements defined for `bb pipeline run --var` flag — what happens with malformed KEY=VALUE input (no = sign, empty key, empty value)? [Coverage, Contract §pipeline run] — **Resolved**: FR-037 now specifies: malformed input (no = sign) rejected with error message.
- [x] CHK037 Are requirements defined for `bb pr diff` output size limits? Large diffs could produce megabytes of stdout — is truncation or streaming behavior specified? [Coverage, Spec §FR-032] — **Resolved**: FR-032 now specifies: CLI streams diff without truncation; consumer handles large output.
- [x] CHK038 Are requirements defined for handling Bitbucket API 5xx server errors? Only 401, 403, 404, 409, 429 are addressed in the plan's error handling section. [Coverage, Plan §Error Handling] — **Resolved**: New FR-048 added for 5xx handling: "Bitbucket server error ({status code}). Try again later."

## Edge Case Coverage

- [x] CHK039 Is the behavior specified when a repository slug contains special characters or URL-unsafe characters? [Edge Case, Spec §FR-009/FR-010] — **Resolved**: FR-010 and edge cases now specify: URL encoding handled by HTTP client; slugs passed as-is.
- [x] CHK040 Is the behavior specified when `bb pr comment add --file <path> --line <num>` references a file not in the PR diff? [Edge Case, Spec §FR-027] — **Resolved**: FR-027 and edge cases now specify: request passed to API; CLI relays any error.
- [x] CHK041 Is the behavior specified when the config file exists but is corrupted or contains invalid JSON? [Edge Case, Spec §FR-003] — **Resolved**: FR-003 now specifies: error with file path and suggestion to re-run bb auth setup.
- [x] CHK042 Is the behavior specified for `bb pipeline logs` when log output is extremely large (the API supports HTTP Range requests — does the CLI use them)? [Edge Case, Spec §FR-042] — **Resolved**: FR-042 now specifies: CLI fetches full log content without Range requests. Logs streamed as plain text.
- [x] CHK043 Is the behavior specified when an OAuth2 access token has expired? Is token refresh in scope, or must the user re-run `bb auth setup`? [Edge Case, Spec §FR-002] — **Resolved**: FR-002 now specifies: token refresh out of scope; error instructs user to obtain new token and re-run auth setup.
- [x] CHK044 Is the behavior specified when the user's Bitbucket App Password lacks the required scopes for a specific operation? (e.g., has `repository:read` but not `pullrequest:write`) [Edge Case, Spec §FR-001] — **Resolved**: FR-001 now documents required scopes per command group and specifies error includes which scope is likely missing.
- [x] CHK045 Is the behavior specified when `bb pr create --reviewer <user>` references a user that doesn't exist or isn't a member of the workspace? [Edge Case, Spec §FR-012] — **Resolved**: FR-012 now specifies: CLI relays the error from the Bitbucket API.

## Security & Credential Requirements

- [x] CHK046 Are requirements defined for preventing credential leakage in `--verbose` mode output? (Auth headers should be masked in HTTP request logs) [Security, Gap] — **Resolved**: FR-008a specifies auth headers masked as "Authorization: Basic ****" or "Authorization: Bearer ****" in verbose output.
- [x] CHK047 Is the requirement for "mode 600" file permissions on the config file portable across platforms? Windows doesn't use Unix file modes — are Windows-equivalent permissions defined? [Security, Spec §FR-003] — **Resolved**: FR-003 now specifies: mode 600 on Unix; user-only ACL (no inheritance) on Windows.
- [x] CHK048 Are requirements defined for what happens if the config directory or file has overly permissive permissions (e.g., world-readable)? Should the CLI warn or refuse to use it? [Security, Gap] — **Resolved**: Edge cases specify: CLI does not check existing permissions; it only sets restrictive permissions when creating/writing.
- [x] CHK049 Are requirements defined for credential handling during `bb auth status` — are credential values masked (showing only first/last characters) or fully hidden? [Security, Gap] — **Resolved**: FR-005a specifies: masked values showing first 4 and last 4 characters with middle replaced by asterisks.
- [x] CHK050 Are the required Bitbucket API scopes per command group documented in the spec? (research.md mentions repository, pullrequest, pipeline scopes but the spec does not) [Security, Gap vs Research] — **Resolved**: FR-001 now documents required scopes: repository, repository:write, pullrequest, pullrequest:write, pipeline, pipeline:write per command group.
- [x] CHK051 Are requirements defined for how the CLI handles HTTPS certificate validation? Should it reject self-signed certificates, or is there a `--insecure` flag? [Security, Gap] — **Resolved**: New FR-050 specifies: HTTPS certificates validated; self-signed rejected; no --insecure flag.

## Agent Integration Requirements

- [x] CHK052 Is the JSON error output format (`{"error": {"message": "...", "status": N}}`) defined in the spec's functional requirements, or only in the contract? [Completeness, Spec §FR-007/FR-008 vs Contract] — **Resolved**: FR-007 now explicitly defines JSON error format in the spec alongside other JSON output structures.
- [x] CHK053 Are requirements defined for JSON output structure when a command succeeds but returns no data (e.g., `bb pr approve` with `--json`)? The contract says `{"success": true, ...}` — are the "..." fields specified? [Clarity, Contract §Output] — **Resolved**: FR-007 and contract now specify action commands return `{"success": true, "message": "..."}` with relevant context fields.
- [x] CHK054 Is the JSON structure for list commands (`{"values": [...], "total": N}`) specified to include pagination metadata (page, pagelen, hasMore) for agents that need to control pagination? [Completeness, Contract §Output] — **Resolved**: FR-007 and contract updated: list commands return `{"values": [...], "page": N, "pagelen": N, "size": N}` with pagination metadata.
- [x] CHK055 Are requirements defined for stable, machine-parseable identifiers in output? (e.g., should PR IDs always appear as integers, pipeline UUIDs always with/without curly braces?) [Clarity, Gap] — **Resolved**: New FR-047a specifies: PR IDs as integers, pipeline UUIDs with curly braces, timestamps in ISO 8601. Contract includes Identifier Formats section.
- [x] CHK056 Are requirements defined for how `--json` interacts with `bb pr diff`? Diff is plain text — does `--json` wrap it in a JSON object, or is `--json` ignored for diff commands? [Gap, Spec §FR-032 vs §FR-007] — **Resolved**: FR-007 now specifies: diff with --json returns `{"diff": "..."}`. Contract updated accordingly.

## Ambiguities & Conflicts

- [x] CHK061 Does the spec define behavior when `bb pr update` is called with `--reviewer` — does it replace all reviewers or add to existing? The contract says "Replace reviewers" but the spec doesn't specify this distinction. [Ambiguity, Spec §FR-014 vs Contract] — **Resolved**: FR-014 now explicitly states "--reviewer flag REPLACES all existing reviewers." Contract updated to "Replace all reviewers."
- [x] CHK062 Is the `--close-source-branch` flag behavior clarified for `bb pr merge`? The flag exists on both `bb pr create` and `bb pr merge` — does the merge flag override the PR's existing setting, or only apply if not set at creation? [Ambiguity, Contract §pr create vs §pr merge] — **Resolved**: FR-018 now states merge flag "overrides the PR's existing close-source-branch setting." Contract updated with "(overrides PR setting)."
- [x] CHK063 Is "clear message" for unsupported task operations (US6 acceptance scenario 3) specified with actual message content, or left to implementation? [Ambiguity, Spec §US6] — **Resolved**: Edge cases section now includes exact message: "Task creation and updates are not supported via the CLI. Use the Bitbucket web UI to manage tasks."
- [x] CHK064 Is the `bb pr list` `--state` filter case-sensitive? The spec uses "open, merged, declined, superseded" (lowercase) in FR-011 but "OPEN, MERGED, DECLINED, SUPERSEDED" (uppercase) in the contract. [Ambiguity, Spec §FR-011 vs Contract] — **Resolved**: FR-011 now specifies case-insensitive acceptance, normalized to uppercase for API. Contract updated with "(case-insensitive)."

## Non-Functional Requirements

- [x] CHK065 Are startup time requirements defined for the CLI binary? (Commander.js was chosen for ~18ms startup, but no NFR captures this target) [Gap, Research vs Spec] — **Resolved**: NFR-005 added: no hard startup time target, but architecture selected to keep startup under 100ms.
- [x] CHK066 Are requirements defined for CLI binary size or installation footprint? [Gap] — **Resolved**: Accepted as out of scope. No binary size requirement — minimal dependency approach inherently keeps footprint small.
- [x] CHK067 Are requirements defined for concurrent usage — can multiple `bb` processes run simultaneously without config file conflicts? [Gap] — **Resolved**: NFR-003 added: config file is read-only during commands (written only by auth setup). No file locking needed.
- [x] CHK068 Are localization or i18n requirements addressed — will error messages and output always be in English, or is multi-language support considered? [Gap] — **Resolved**: NFR-001 added: English only, i18n out of scope.
- [x] CHK069 Are requirements defined for signal handling (Ctrl+C / SIGINT) during long-running operations like paginated fetches or pipeline log streaming? [Gap] — **Resolved**: NFR-002 added: default Node.js signal handling; SIGINT terminates immediately; no cleanup of in-flight requests.
- [x] CHK070 Are backward compatibility requirements defined for the config file format? If the config schema changes in future versions, how is migration handled? [Gap] — **Resolved**: NFR-004 added: config format not versioned; migration via re-running bb auth setup.
